# campnw Post-v1.0 Feature Requirements

**Date:** March 28, 2026
**Scope:** v1.1 "Better Search + Coverage" and v1.2 "Trips + Watches"
**Audience:** Solo developer (you) — sized for one person working full-time

---

## v1.1 "Better Search + Coverage"

---

### 1. Natural Language Search (A1)

**Size:** M (3-5 days)

**Description:** A freeform text input ("dog-friendly lakeside spot near Portland, July 4th weekend") that Haiku parses into structured search parameters and executes the existing search pipeline. Serves as the collapsed/quick-entry mode for the search form — power users keep structured filters, casual users type a sentence.

**Requirements:**
- Text input field above or replacing the collapsed search form; typing submits to Haiku for structured extraction via tool_use (JSON schema matching existing search API params)
- Parsed interpretation displayed before search executes ("Searching for: lakeside, dog-friendly near Portland, Jul 3-5") with an "Edit filters" link that opens the structured form pre-filled
- Date inference handles relative expressions ("this weekend", "next month", "July 4th weekend", "late August") with correct year assumption
- Tag mapping expands user vocabulary to existing tags (e.g., "waterfront" maps to lakeside/riverside/beach; "quiet" maps to walk-in/primitive)
- Falls back to structured form with a toast if Haiku returns low-confidence or unparseable input (no silent wrong searches)
- Latency budget: parsing completes in under 1.5s; if slower, show a brief "Understanding your search..." indicator

**Dependencies:** Existing search API, Anthropic API key in production env (already present for trip planner)

**Key risk:** Date inference edge cases — "4th of July weekend" in March 2026 should resolve to July 2026, not create an invalid past date. Build a test suite of 20-30 natural language queries with expected structured output before writing the prompt.

### 2. Registry Expansion (MT, WY, NorCal via RIDB)

**Size:** S (1-2 days)

**Description:** Expand the campground registry from 3 states (WA, OR, ID) to include Montana, Wyoming, and Northern California via the existing RIDB seed pipeline. Broadens discovery coverage for road-trippers and shoulder-PNW destinations without new provider integrations.

**Requirements:**
- Seed script adds MT, WY, and CA campgrounds from RIDB (CA filtered to facilities north of ~38N latitude — roughly Yosemite and above)
- All new campgrounds get drive-time-from-Seattle calculated and stored (existing geo pipeline)
- Run tag enrichment (existing `enrich` CLI) on new campgrounds before shipping
- State filter UI updated to include new states; source filter unchanged (all are recgov)
- Registry total documented in CLAUDE.md after seeding
- Search results for new states work identically to existing WA/OR/ID results

**Dependencies:** RIDB API key, existing seed_registry.py script

**Key risk:** NorCal latitude filtering — need to decide the cutoff. Suggest 38.5N (includes Yosemite, Lassen, Shasta; excludes Bay Area urban parks). Verify RIDB returns latitude for CA facilities.

---

### 3. Tag Taxonomy Audit (C3)

**Size:** XS (<1 day)

**Description:** One-shot Sonnet analysis of the full tag vocabulary (29 tags) to identify merges, gaps, and overly-broad tags. Improves search filtering accuracy and recommendation quality by cleaning up the taxonomy that feeds personalization.

**Requirements:**
- Export full tag data: tag name, frequency count, 3-5 sample campground names per tag
- Single Sonnet call with taxonomy context; output is a structured recommendation (proposed merges, additions, removals with rationale)
- Developer reviews output manually and decides which changes to apply
- If taxonomy changes, re-run existing `enrich` CLI for affected campgrounds
- Document the final taxonomy decisions (what changed and why) in a short changelog

**Dependencies:** None

**Key risk:** None meaningful. Output is advisory; no automated changes. The only cost is re-running enrichment for affected campgrounds if tags change (~$0.10).

---

### 4. Registry Description Rewrite (C1)

**Size:** S (1-2 days)

**Description:** Replace bureaucratic RIDB descriptions with user-friendly campground summaries generated by Haiku. Each campground gets a 1-sentence elevator pitch (for result cards), a 2-3 sentence "what makes this place special" description (for expanded view), and a "best for" label.

**Requirements:**
- New registry columns: `elevator_pitch TEXT`, `description_rewrite TEXT`, `best_for TEXT`
- Batch CLI command (clone existing `enrich` pattern) processes all campgrounds; idempotent (skip already-rewritten)
- Elevator pitch renders on result cards (before expand); full description and best_for render in expanded view
- Review a sample of 20-30 outputs before bulk-writing; reject and re-prompt any that hallucinate amenities not in the source data
- Graceful degradation: if rewrite columns are NULL, fall back to original RIDB description
- Run for new MT/WY/NorCal campgrounds added in feature #2

**Dependencies:** Tag Taxonomy Audit (C3) should run first so descriptions reference the cleaned taxonomy

**Key risk:** Hallucinated amenities — Haiku may infer "lakeside" from a description that mentions a lake 5 miles away. Prompt must constrain to stated facts only. Spot-check is the mitigation.

### 5. Post-Search Result Summarizer (A2)

**Size:** S (1-2 days)

**Description:** After search results stream in via SSE, generate a 2-3 sentence AI summary that surfaces patterns across all results — where availability clusters, weekend vs. weekday tradeoffs, standout options. Helps users make sense of 10+ result cards without scanning each one.

**Requirements:**
- Triggers only when result count > 5 (small result sets are self-evident)
- Summary sent as a trailing SSE event (new event type, e.g., `event: summary`) after all result cards
- Summary card renders above results (pinned) or as a dismissible banner at the top
- Haiku call includes: aggregated result data (campground names, available date counts, tags, drive times) and the user's original search params
- Latency: summary arrives within 2s of last result card; total search experience not gated on summary
- If Haiku call fails or times out (3s), no summary shown — no error state, just absent

**Dependencies:** Existing SSE search stream, Anthropic API key

**Key risk:** Latency stacking — if search itself takes 4-5s and summary adds 2s, total wait is 6-7s. Mitigated by rendering summary after results (progressive enhancement, not blocking). Monitor P95 of summary generation time.

---

### 6. Personalized Recommendation Reasons (A5)

**Size:** S (1-2 days)

**Description:** Replace template-based recommendation reasons ("Similar tags to your recent searches") with LLM-generated contextual explanations that reference the user's specific search patterns and the campground's unique attributes.

**Requirements:**
- For each batch of recommendations (up to 5), send to Haiku: user's recent search patterns (top tags, states, date ranges from search_history), campground data (tags, vibe description, drive time), and affinity scores
- Generated reasons cached in SQLite for 24 hours per user; cache key is user_id + recommendation set hash
- Graceful degradation: if Haiku unavailable or cache miss + timeout, fall back to existing template strings
- Reasons are 1-2 sentences max; must reference at least one specific attribute of the recommended campground
- No recommendation reasons generated for logged-out users (no search history to reference)

**Dependencies:** Existing recommendation engine (`get_recommendation_affinities`), search_history data

**Key risk:** Cold-start users with <3 searches get generic reasons that feel worse than templates. Set a minimum search_history threshold (3+ searches) before enabling LLM reasons; below that, keep templates.

---

### 7. Search Analytics Digest (B1)

**Size:** S (1-2 days)

**Description:** Weekly automated job that aggregates search_history data and produces a structured product intelligence report: most-requested regions/dates with zero availability, underserved tag combinations, week-over-week trend changes, and registry gap identification.

**Requirements:**
- APScheduler job runs weekly (Sunday 11pm Pacific), reusing existing scheduler infrastructure
- Aggregates search_history: group by state, tags, date_range, result_count, source; identifies zero-result searches and popular filter combos
- Sends aggregated summary (not raw rows) to Haiku; output is structured JSON
- Stores report in new `analytics_digests` SQLite table (id, created_at, report_json, summary_text)
- Optionally pushes a 3-5 bullet summary via ntfy (reuse existing ntfy integration)
- Handles sparse data gracefully: if <10 searches in the week, skip generation with a log message

**Dependencies:** APScheduler (exists), search_history table (exists), ntfy integration (exists)

**Key risk:** Small user base means small sample sizes. Reports may be dominated by a single user's patterns. Acknowledge this in the report template ("Based on N searches from M users this week").

---

### 8. Dark Mode Heatmap Fix

**Size:** XS (<1 day)

**Description:** Fix the known issue where heatmap levels 0 and 1 are nearly indistinguishable in dark mode. Adjust the dark mode heatmap color scale to ensure all 5 levels are visually distinct.

**Requirements:**
- All 5 heatmap levels (`--heatmap-0` through `--heatmap-4`) have minimum 2:1 contrast ratio between adjacent levels in dark mode
- Colorblind-safe: maintain single-hue progression (do not introduce red-green differentiation)
- Update dark mode overrides in tokens.css; light mode values unchanged
- Legend (if present) reflects new values
- Visual verification in both themes before shipping

**Dependencies:** None

**Key risk:** None. Pure CSS change with no behavioral impact.

## v1.2 "Trips + Watches"

---

### 1. Trip Object

**Size:** L (1-2 weeks)

**Description:** A Trip is a first-class entity that groups campgrounds, date ranges, and watches into a single plannable unit. Users can save campgrounds to a trip from search results, see aggregated availability across all campgrounds in the trip, and manage associated watches from one view. This is the foundation for features 3 and 4 in this milestone.

**Requirements:**
- New `trips` table: id, user_id, name, start_date, end_date, notes, created_at, updated_at
- New `trip_campgrounds` junction table: trip_id, facility_id, source, added_at
- CRUD API endpoints: POST/GET/PUT/DELETE `/api/trips`, POST/DELETE `/api/trips/{id}/campgrounds`
- "Save to trip" button on search result cards; if user has no trips, prompt to create one inline
- Trip detail view (`/trips/{id}`): lists all campgrounds with their current availability for the trip's date range, aggregated calendar heatmap across all campgrounds
- Watches created from a trip are linked (trip_id on watches table); trip view shows watch status per campground
- Trips are user-owned; require authentication. Unauthenticated users see "Sign in to save trips"
- Maximum 10 trips per user (free), 25 (Pro)

**Dependencies:** User auth (exists), watch system (exists)

**Key risk:** Schema design — the trip_campgrounds junction table needs to handle campgrounds from different sources (recgov vs wa-state vs or-state) with different facility_id formats. Use composite key (facility_id + source) rather than facility_id alone.

---

### 2. Multi-Campground / Template Watches

**Size:** L (1-2 weeks)

**Description:** Watches that match a search pattern rather than a single campground. "Alert me when any lakeside campground within 2 hours of Seattle has a Friday-Saturday opening in July" — the system expands the template to matching registry campgrounds and polls them as a group.

**Requirements:**
- New watch type: `template` (vs existing `single`); stored with search params (tags, state, max_drive, from_location, source) instead of facility_id
- Template watch expands to matching campgrounds at poll time (not creation time) — so newly-added registry entries are automatically included
- Notification groups results by campground ("3 new openings: Kalaloch (2 sites), Moran (1 site)")
- Template watches count as 1 watch toward the user's watch limit, regardless of how many campgrounds they expand to
- Poll budget: template watches poll a maximum of 20 campgrounds per cycle to avoid API rate limit issues; if template matches more, poll the 20 closest to from_location
- UI: new "Watch this search" button on search results page (in addition to per-campground "Watch this" on cards)
- Template watch edit allows adjusting search params after creation

**Dependencies:** Trip Object (feature #1) for trip-linked template watches; existing watch polling infrastructure

**Key risk:** Rate limiting — a template watch matching 50 campgrounds could consume the entire poll budget. The 20-campground cap per template is critical. Need to also consider interaction with Pro 5-minute polling: 20 campgrounds every 5 minutes is 240 API calls/hour per template watch.

---

### 3. Watch Sharing

**Size:** S (1-2 days)

**Description:** Generate a shareable link for a watch or trip that lets a friend view the same availability without creating an account. Useful for group trip planning — "here's what I'm monitoring, take a look."

**Requirements:**
- Shareable link via UUID token: `/shared/{uuid}` resolves to a read-only view of the watch/trip's current availability
- Shared view shows: campground name(s), date range, current availability snapshot, booking links. Does not show the owner's identity
- Links expire after 30 days; creator can revoke at any time
- No account required to view a shared link
- Share button on watch cards and trip detail view; copies link to clipboard with confirmation toast
- Shared watches are read-only — viewer cannot modify, pause, or delete

**Dependencies:** Trip Object (feature #1) for trip sharing; watch system (exists)

**Key risk:** Abuse vector — shared links trigger availability checks on view. Rate limit shared link views to 10/hour per UUID to prevent scraping via shared links.

---

### 4. Trip Planner to Persistent Itinerary

**Size:** M (3-5 days)

**Description:** Connect the conversational trip planner (`/plan`) to the Trip object system. When the AI recommends campgrounds, users can save the output as a Trip with one click, pre-populated with the discussed campgrounds and dates. Bridges the gap between conversational exploration and actionable planning.

**Requirements:**
- "Save as Trip" button appears in chat when the AI has recommended specific campgrounds (detect via tool_use results in conversation)
- Clicking creates a Trip object pre-populated with: campgrounds from the conversation's search/check tool calls, date range inferred from conversation, AI-generated trip name
- Saved trip appears in `/trips` with a "Created from Trip Planner" badge
- User can edit the trip after saving (add/remove campgrounds, adjust dates)
- If user is not authenticated, "Save as Trip" prompts sign-in first (then saves)
- One conversation can produce multiple trips (e.g., "Plan A: beach, Plan B: mountains")

**Dependencies:** Trip Object (feature #1), existing trip planner (exists)

**Key risk:** Extracting structured trip data from conversation history — the planner's tool_use calls contain facility_ids and dates, but mapping them to a coherent trip requires parsing the conversation's intent (which campgrounds were final recommendations vs. rejected options). Start simple: save all campgrounds the AI checked/recommended, let the user prune.

---

### 5. Onboarding + Profile Improvements

**Size:** S (1-2 days)

**Description:** Improve the new-user experience with a lightweight onboarding flow and richer profile settings. Currently, sign-up drops users back to search with no guidance; home base and preferences are buried in settings.

**Requirements:**
- Post-signup onboarding: 2-step modal — (1) set home base location (city/zip with geocode), (2) select preferred tags (checkboxes from tag taxonomy, max 5). Skip button on each step
- Home base and preferred tags save to user profile; pre-fill search form defaults
- Profile page shows: home base (editable), preferred tags (editable), default trip type, notification preferences, subscription status
- Onboarding modal only shows once (flag in user record); returning users go straight to search
- Preferred tags feed into recommendation affinity scoring (boost campgrounds matching preferred tags)

**Dependencies:** User auth (exists), tag taxonomy (ideally after C3 audit)

**Key risk:** Over-engineering onboarding for a small user base. Keep it to 2 steps max. If users skip both steps, the experience should be identical to today — no degradation.

---

### 6. Campground Comparison (A3)

**Size:** S (1-2 days)

**Description:** Select 2-3 campgrounds from search results and get a structured AI comparison highlighting availability overlap, distance tradeoffs, vibe differences, and a recommendation based on the user's search criteria.

**Requirements:**
- "Compare" checkbox appears on result cards; "Compare selected" button enables when 2-3 campgrounds are checked
- Comparison view (modal or inline panel) shows: side-by-side data table (drive time, available dates, tags, source) and a Haiku-generated narrative comparison (2-4 sentences)
- Haiku prompt includes user's search params as context for the recommendation ("If weekends matter most, X is better because...")
- Comparison dismissible; does not navigate away from search results
- Works with campgrounds from mixed sources (recgov + wa-state in same comparison)
- If Haiku call fails, show the data table without the narrative — still useful

**Dependencies:** Search results (exists), Anthropic API key

**Key risk:** Low. Informational feature with graceful degradation. Main UX question: modal vs. inline panel. Recommend inline panel below the search results to avoid losing scroll position.

---

### 7. Historical Pattern Extraction (C2) — "Booking Tips"

**Size:** M (3-5 days)

**Description:** Analyze accumulated availability_history data to extract per-campground booking patterns, rendered as "Booking Tips" in expanded campground cards. Example: "Ohanapecosh fills weekends 6 months out. Weekday cancellations appear 2-3 weeks before the date." Validates whether polling history is rich enough for v1.3's statistical model.

**Requirements:**
- Data pipeline: aggregate availability_history into per-campground summaries (availability rate by day-of-week, cancellation timing distribution, seasonal fill patterns)
- Only generate tips for campgrounds with 30+ days of polling observation; show "Still learning..." for sparse data
- Per-campground Haiku call converts aggregated stats into 2-4 human-readable booking tips
- Tips stored in registry (new `booking_tips TEXT` column); refreshed monthly via batch job
- Rendered in expanded card view below vibe description, under a "Booking Tips" heading
- Include data freshness indicator ("Based on data through March 2026")
- Batch job is idempotent and skips campgrounds with <30 days of data

**Dependencies:** availability_history table (exists, collecting since March 22, 2026). By v1.2 timeframe (~Q4 2026), should have 6+ months of data for actively-polled campgrounds. Sparse data for campgrounds that have never been watched.

**Key risk:** Data sparsity — availability_history only collects data for campgrounds that are actively watched or searched. Popular campgrounds will have rich data; obscure ones will have nothing. The 30-day threshold is critical to avoid generating nonsense tips from 3 data points.

---

### 8. Notification Quality Feedback Loop (B2)

**Size:** S (1-2 days)

**Description:** Monthly batch job that correlates notification messages with user actions (booking clicks within 1 hour of notification) to identify which notification patterns drive engagement. Outputs prompt adjustment suggestions for the notification enrichment system.

**Requirements:**
- Join notification_log with event tracking (book_click events) using a 1-hour attribution window
- Build dataset: notification_text, urgency_score, campground, time_of_day, clicked (boolean)
- Feed batch to Haiku: identify patterns correlating with booking clicks
- Store analysis in `analytics_digests` table (reuse from B1) with type=notification_quality
- Output includes: specific prompt adjustment suggestions for notification enrichment, labeled as suggestions (not auto-applied)
- Minimum threshold: only run when 50+ notifications have been sent in the period; below that, log "insufficient data" and skip

**Dependencies:** notification_log table (exists), event tracking (exists), analytics_digests table (from B1)

**Key risk:** Small sample size is the dominant concern. With a handful of users, the signal-to-noise ratio will be poor. Set the 50-notification threshold and treat early outputs as directional, not prescriptive. Value increases significantly if user base grows.

### v1.1 Total Effort: ~3-4 weeks

| # | Feature | Size | Days |
|---|---------|------|------|
| 1 | Natural Language Search | M | 3-5 |
| 2 | Registry Expansion (MT/WY/NorCal) | S | 1-2 |
| 3 | Tag Taxonomy Audit | XS | <1 |
| 4 | Registry Description Rewrite | S | 1-2 |

**Suggested sequencing:** 3 (taxonomy) then 4 (descriptions) then 2 (expansion + enrich new entries). These are batch/infra. Then 1 (NL search) as the headline feature. Then 5, 6, 7 in any order. 8 whenever.

| # | Feature | Size | Days |
|---|---------|------|------|
| 1 | Trip Object | L | 5-10 |
| 2 | Multi-Campground / Template Watches | L | 5-10 |
| 3 | Watch Sharing | S | 1-2 |
| 4 | Trip Planner to Persistent Itinerary | M | 3-5 |
| 5 | Onboarding + Profile Improvements | S | 1-2 |
| 6 | Campground Comparison | S | 1-2 |
| 7 | Historical Pattern Extraction | M | 3-5 |
| 8 | Notification Quality Feedback Loop | S | 1-2 |

**Suggested sequencing:** 1 (Trip Object) first — it's the foundation for 3 and 4. Then 2 (template watches) can parallel with 4 (planner-to-trip). 5 and 6 are independent. 7 depends on data maturity, not code — start the pipeline early. 8 last (needs accumulated notification data).

**Critical path:** Trip Object (1) blocks Watch Sharing (3) and Planner-to-Itinerary (4).
